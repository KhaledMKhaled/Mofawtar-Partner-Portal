==================================================
Greenfield Unified Financial Architecture
==================================================

Use one unified financial ledger called:

financial_items

financial_items is the single source of truth for all financial records.

Do not create separate source-of-truth tables for:
- order payments
- partner commissions
- sales commissions
- payout batches

Pages like Payments, Partner Commissions, and Sales Commissions should be filtered views of financial_items.

Examples:
- Payments page = financial_items where type = payment_item
- Partner Commissions page = financial_items where type = partner_commission_item
- Sales Commissions page = financial_items where type = sales_commission_item

==================================================
1. Financial Item Types
==================================================

Each financial item must have one type.

Allowed types only:

1. payment_item
Arabic: طلب مدفوعات
Purpose:
Represents the net amount that the partner must pay to the company after deducting partner commission.

2. partner_commission_item
Arabic: طلب عمولة شريك
Purpose:
Represents the partner commission amount that was deducted from the customer payment and must be documented and settled as an offset, not as a cash payout from company to partner.

3. sales_commission_item
Arabic: طلب عمولة مبيعات
Purpose:
Represents the Sales user commission that the company will pay separately to the Sales user.

==================================================
3. Financial Items Created on Activation
==================================================

When a request becomes Activated, automatically create financial items.

For every Activated request, create:

1. payment_item
Amount = net amount due to company

2. partner_commission_item
Amount = partner commission amount

3. sales_commission_item
Amount = sales commission amount
Only create this if sales commission is enabled and amount is greater than zero.

All items start with:

status = not_added_to_claim

Calculation example:

Item price before tax = 1000
Tax percentage = 14%
Final price after tax = 1140
Partner commission = 20% of 1000 = 200
Sales commission = 5% of 1000 = 50
Net amount due to company = 1140 - 200 = 940

Create:

payment_item:
amount = 940

partner_commission_item:
amount = 200

sales_commission_item:
amount = 50

Important:
Partner commission settlement is not a company-to-partner cash payout.
It is documentation and approval of the commission amount already deducted by the partner.

==================================================
5. Eligibility Rules
==================================================

Do not create extra statuses for safety period or payout cycle.

Use eligibility fields.

Payment Item:
- Usually claimable immediately after activation.
- eligible_for_claim_at = activation date
- is_claimable = true after activation unless voided or blocked.

Partner Commission Item:
- Claimable only after partner safety period ends.
- Safety period starts from activation date.
- eligible_for_claim_at = activation date + partner safety period days.

Before eligible date:
- status remains not_added_to_claim
- is_claimable = false
- claim_block_reason = Inside safety period

After eligible date:
- status remains not_added_to_claim
- is_claimable = true

Sales Commission Item:
- Claimable based on sales commission payout cycle.

Monthly payout:
- Any sales commission generated in a calendar month becomes claimable on the first day of the next month.

Example:
Activation date = May 10
Monthly eligible_for_claim_at = June 1

Quarterly payout:
- Any sales commission generated inside a calendar quarter becomes claimable on the first day after the quarter ends.

Example:
Activation date = May 10
Quarter = April to June
Quarterly eligible_for_claim_at = July 1

Optional future setting:
- payout_delay_days

Before adding any financial item to a claim, validate:
- item.status = not_added_to_claim
- item.is_voided = false
- current date >= eligible_for_claim_at
- item.type matches claim type
- related request is Activated
- user has permission

8. Claim Items

- claim_id
- financial_item_id

- commission_base_snapshot
- tax_snapshot
- net_due_snapshot
- created_at

Rules:
- claim_items must reference financial_items.id only.
- A financial item can be in only one active claim at a time.
- When item is added to claim:
  - item.status = added_to_claim
  - item.claim_id = claim.id
  - create claim_items record
  - save amount_snapshot
  - update claim total amount
  - create audit log
  - update Customer 360 Timeline

If claim is rejected before settlement:
- claim.status = rejected
- related items return to not_added_to_claim
- related item claim_id is cleared
- rejection reason is required
- audit log is created
- Customer 360 Timeline is updated

9. Settlement Types
==================================================

Use one settlements table with a type field.

Allowed settlement types:

1. payment_settlement
Arabic: تسوية استلام مستحق الشركة
Purpose:
Confirm the company received the net amount due from the partner.

2. partner_commission_settlement
Arabic: تسوية اعتماد خصم عمولة الشريك
Purpose:
Confirm that the partner commission deduction has been documented and settled.

3. sales_commission_settlement
Arabic: تسوية صرف عمولة المبيعات
Purpose:
Confirm that Sales commission payout has been completed.

Settlement fields:
- id
- settlement_number
- type
- status
- total_amount
- created_by
- completed_by
- created_at
- completed_at
- cancelled_at
- cancellation_reason
- notes

Settlement statuses:
- draft
- completed
- cancelled

Relationship rule:
One settlement can include one or more claims of the same type.

Recommended implementation:
- claims.settlement_id nullable
- one settlement has many claims
- each claim belongs to one settlement only

Do not use settlements.claim_id as a required one-to-one relation.

When a claim is added to settlement:
- claim.settlement_id = settlement.id
- claim.status = in_settlement
- all related financial_items.status = added_to_settlement
- all related financial_items.settlement_id = settlement.id
- create audit log
- update Customer 360 Timeline

When settlement is completed:
- settlement.status = completed
- all related claims.status = settled
- all related claims.settled_at = now
- all related financial_items.status = settled
- all related financial_items.settled_at = now
- create audit log
- update Customer 360 Timeline

11. Void, Refund, Cancel, and Adjustment
==================================================

Do not add extra lifecycle statuses for refund or cancel.

Use void fields.

If item is cancelled before claim:
- keep status = not_added_to_claim
- set is_voided = true
- set void_type
- set void_reason
- set voided_at
- set voided_by

If item is already in claim but not settlement:
- reject or adjust the claim
- return item to not_added_to_claim if needed
- set is_voided = true when applicable
- reason is required

If item is already settled:
- do not edit the settled item
- create a negative adjustment financial_item
- set is_adjustment = true
- link adjustment_for_item_id to original item
- amount should be negative
- require adjustment reason
- log audit entry
- show it in Customer 360 Timeline

- Reports

Pending if sales_commission_item exists and is not settled.
Closed if:
- sales_commission_item.status = settled

Open if any related financial item is not settled.
Fully Settled if all related financial items are settled.
